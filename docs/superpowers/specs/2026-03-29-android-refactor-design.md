# 安卓化重构设计文档

## 1. 背景

当前项目是一个基于 `React + TypeScript + Vite` 的前端应用，同时仓库中保留了 `src-tauri` 桌面端相关结构。项目已经具备以下核心能力：

- HRT 相关剂量记录与管理
- 实验室结果记录
- 基于现有药代动力学逻辑的模拟与图表展示
- 多语言、主题切换、本地导入导出

本次目标不是重写整套产品，而是在尽量复用现有前端与核心逻辑的前提下，将项目重构为一个可安装、可运行的安卓 App 首版。

## 2. 本次改造目标

本次改造的明确目标如下：

- 将现有项目接入安卓容器，产出可安装的 `APK`
- 最大化复用现有 `React` 页面、状态管理与计算逻辑
- 首版仅保留纯本地能力，不接入账号体系与云同步
- 保留现有 Web/Tauri 代码，不做破坏性删除
- 让应用在安卓手机上具备稳定的页面展示和基础交互体验

## 3. 非目标

以下内容不属于本次首版安卓化范围：

- 不进行 `Kotlin + Jetpack Compose` 原生重写
- 不在首版中保留账号登录、云保存、云读取、管理员页
- 不追求一次性完成正式应用市场发布所需的全部素材与发布配置
- 不进行与安卓化无关的大规模业务重构

## 4. 技术路线选择

本次采用 `Capacitor + 现有 React 前端` 作为推荐和实施路线。

选择原因如下：

- 与当前技术栈兼容度高，接入成本低
- 可以直接复用现有 `vite build` 产物
- 更适合“尽快产出可安装 APK”的目标
- 后续如果需要补充原生能力，可以逐步引入 Capacitor 插件或自定义桥接

未选择的路线：

- `Tauri 2 Mobile`：迁移成本和不确定性更高，不适合当前“快速得到安卓版”的目标
- 自定义 `WebView` 壳：可维护性与工程完整性不如 Capacitor
- 原生安卓重写：工作量过大，不符合当前范围

## 5. 总体架构设计

重构后的项目结构保持“一个前端，多端承载”的思路：

- `src/`：继续作为主要前端业务代码来源
- `logic.ts`、`worker.ts` 等核心逻辑文件：继续复用
- `android/`：新增 Capacitor 生成的安卓工程
- `src-tauri/`：保留，但不参与本次安卓首版交付

整体架构关系如下：

1. 前端仍由 `Vite` 负责构建静态资源
2. Capacitor 将构建后的前端资源装入安卓应用容器
3. 安卓 App 通过 WebView 运行现有前端页面
4. 平台相关能力通过统一适配层收口，而不是散落在页面组件中

## 6. 功能范围设计

### 6.1 安卓首版保留功能

- 首页模拟结果展示
- 历史剂量记录与编辑
- 实验室结果录入与编辑
- 图表展示
- 主题切换
- 语言切换
- 本地数据导入导出
- 免责声明、本地设置、体重设置等纯本地功能

### 6.2 安卓首版下线功能

- `Account` 页面
- 登录弹窗与账号相关操作
- 云保存
- 云读取
- 管理员页面

### 6.3 下线策略

下线功能不做粗暴删除，而是采用“能力开关 + UI 隐藏”的方式处理：

- 安卓环境中隐藏对应导航入口
- 不向用户暴露不可用按钮
- 尽量保留原有代码结构，方便后续恢复

## 7. 数据存储设计

安卓首版优先保持与现有数据模型兼容，避免因为存储层切换影响业务稳定性。

### 7.1 首版策略

- 优先继续使用现有前端本地存储方案
- 如果当前主要依赖 `localStorage`，则安卓版继续复用
- 不在首版强行迁移到 SQLite、Room 或其他原生存储方案

### 7.2 原因

- 当前目标是低风险迁移
- 现有数据结构和页面读写逻辑可以最大限度复用
- 存储改造会牵连大量状态管理与导入导出逻辑，不适合首版

### 7.3 后续演进空间

如果未来需要更高可靠性，可以新增统一存储服务层，再逐步替换底层实现，例如：

- `localStorage` 抽象层
- Capacitor Preferences
- 文件存储
- SQLite

## 8. 平台适配设计

为了避免业务组件中出现大量平台判断，本次会新增统一平台适配层。

### 8.1 适配层职责

- 判断当前运行环境：Web、Tauri、Android 容器
- 封装剪贴板、分享、导出等平台相关能力
- 为未来增加原生文件系统能力预留接口

### 8.2 首版重点适配内容

- 导出行为
- 复制到剪贴板
- 可能涉及下载文件的逻辑
- 安卓环境能力开关

### 8.3 设计原则

- 页面组件只调用统一服务，不直接依赖平台 API
- 先做可用，再做增强
- 安卓不支持或体验不佳的能力要有优雅降级方案

## 9. 导入导出设计

导入导出是本项目的关键本地能力，安卓首版必须保留。

### 9.1 导入

- 保留现有 JSON 导入逻辑
- 兼容加密数据与压缩数据的现有处理流程
- 优先复用已有业务实现

### 9.2 导出

由于浏览器中的“下载文件”在安卓容器环境下体验可能不理想，导出采用分层策略：

1. 第一优先：通过原生分享或保存能力导出 JSON
2. 第二优先：复制导出文本到剪贴板
3. 兜底方案：保留现有浏览器下载逻辑，作为兼容回退

## 10. 界面与交互设计

现有项目已经具备一定移动端基础，因此首版以“移动端稳定化”而不是“重做 UI”为主。

### 10.1 保留的页面结构

- `Home`
- `History`
- `Lab`
- `Settings`

### 10.2 隐藏的页面结构

- `Account`
- `Admin`

### 10.3 重点处理问题

- 刘海屏与底部手势区域安全区
- 弹窗在小屏设备上的高度、滚动和关闭体验
- 触屏下日期时间选择体验
- 软键盘弹起导致的布局挤压与滚动错乱
- 图表在安卓 WebView 下的渲染稳定性

### 10.4 视觉策略

- 尽量保留现有视觉语言
- 优先解决可用性与稳定性
- 不做与首版目标无关的大范围视觉翻新

## 11. 项目结构设计

为了满足“保留现有 Web/Tauri 代码，同时新增安卓工程”的要求，建议结构如下：

- `src/`：现有前端页面与逻辑
- `src-tauri/`：保留
- `android/`：新增安卓工程
- `docs/superpowers/specs/`：设计与计划文档
- 适配层代码放入 `src/services/` 或新增 `src/platform/`

## 12. 实施步骤设计

### 阶段一：安卓工程接入

- 安装并配置 Capacitor
- 新增安卓工程
- 将前端构建产物接入 Capacitor
- 增加安卓运行和构建脚本

### 阶段二：平台适配层建设

- 增加运行环境判断
- 抽离导出、分享、剪贴板等平台能力
- 为安卓首版提供兼容实现

### 阶段三：功能裁剪

- 隐藏账号与管理员相关入口
- 防止不可用逻辑在安卓环境中被触发
- 保持现有本地功能可访问

### 阶段四：移动端稳定性修复

- 调整安全区和布局
- 调整弹窗和表单
- 修复安卓交互问题

### 阶段五：打包验证

- 生成首版安卓安装包
- 验证核心页面与核心交互
- 确认本地数据流程可用

## 13. 风险与缓解策略

### 风险一：浏览器下载逻辑在安卓容器中表现不一致

缓解：

- 提前抽离导出逻辑
- 增加分享与剪贴板回退

### 风险二：WebView 环境下部分浏览器 API 兼容性不足

缓解：

- 引入平台适配层
- 对关键 API 做能力检测和降级处理

### 风险三：移动端小屏表单与弹窗体验不稳定

缓解：

- 将移动端稳定性作为独立阶段处理
- 在关键页面上优先验证输入、滚动、关闭路径

### 风险四：直接改动过多导致 Web 端回归

缓解：

- 保持平台差异通过适配层和能力开关控制
- 尽量减少对业务计算逻辑的侵入式修改

## 14. 验收标准

完成本次安卓首版后，应满足以下标准：

- 项目可以构建出可安装的安卓 `APK`
- 安卓环境可正常进入应用并浏览核心页面
- 首页模拟与图表展示正常
- 历史记录与实验室结果可新增、编辑、删除
- 设置、语言、主题等本地功能可用
- 导入导出至少具备一条稳定可用路径
- 账号、云同步、管理员功能不会在安卓版中误暴露
- 现有 Web/Tauri 代码结构仍然保留

## 15. 结论

本次重构采用“复用现有前端 + 新增 Capacitor 安卓容器”的策略，是当前范围、成本、交付速度和风险控制之间最平衡的方案。

该方案的核心价值在于：

- 以最小重写成本产出首版安卓 App
- 尽量保留现有业务逻辑与界面资产
- 避免在首版中引入账号、云同步、原生重写等高复杂度事项
- 为后续逐步增强原生能力预留清晰的技术演进空间
